# 表格页面设计规范

后续所有表格页面设计和改动，都必须优先遵循本规范。

## 视觉与对齐

- 表格内容默认居中，包括表头、单元格、输入框、选择框、状态徽章。
- 长文本列可以换行，但单元格整体仍保持居中，不回退为默认左对齐。
- 需求名称、说明、建议操作、验收标准等长文本允许 `word-break` / `overflow-wrap`，但视觉重心保持居中。
- 长文本可编辑字段优先使用 textarea，不要用单行 input 挤压内容。
- 表格按钮字体、字号、高度要与同一工具栏内其他按钮一致。
- 必填字段使用红色 `*` 标识，建议通过统一样式如 `.required-label::after` 实现。

## 横向滚动

- 表格横向滚动条必须靠近当前表格，而不是只出现在页面底部。
- 每个表格卡片或 `.mini-table` 自己负责横向滚动：`overflow: auto`。
- 需求池、人员配置这类基础维护清单不要放在限定高度的表格框内，表格高度随内容自然展开，只保留横向滚动。
- 如果基础维护清单行数较多导致底部横向滚动条需要滚到页面很下方才能看到，必须在表格上方提供一个可见的横向滚动条/同步滚动控件，让用户不必到表格底部也能左右拖动。
- 风险明细、预算执行明细等分析类表格可设置适度 `max-height`，但必须保证横向滚动条能在当前卡片内看到。
- 对明细类大表，优先拆成多个较短表格或分区卡片。
- 展开详情行不得被第一列样式限制宽度；`colSpan` 详情单元格需要允许完整内容参与横向滚动。

## 需求池表格

- 需求池是基础数据维护页，不展示风险/预算执行结论。
- 提交需求按钮必须与表格操作按钮分组展示：提交需求单独放在需求清单/表格左上方，清除筛选、字段筛选、恢复默认列顺序、导入、导出等表格操作放在右上方。
- 新增字段时同步更新：
  - `demandColumnLabels`
  - 默认列顺序
  - 单元格编辑控件
  - 展开详情表单
  - CSV 导入、导出、模板
- 列顺序要符合业务阅读顺序。人员/角色字段靠近提出人，验收标准靠近类型/需求说明区域，预算字段放在预算状态附近，工作量字段放在预估人天附近。
- 需求池中的验收标准是长文本字段，表格列宽应明显宽于普通枚举/日期列（建议 36ch 起），详情/提交表单中应跨 2 列或使用更宽区域。
- 需求清单表头的所有字段标签必须单行显示，不允许换行；通过横向滚动承载宽度。
- 需求清单表头的所有必填字段必须带红色 `*`，与提交/详情表单保持一致。
- 需求方只能用下拉框，不允许在需求池或提交需求页面自由输入；如历史数据存在非标准值，只能作为该行兼容选项临时展示。
- 需求单号必须唯一，提交、编辑、导入都要校验重复。
- 需求池所有列必须支持排序。
- 排序应在筛选后执行，再进入渲染。
- 数字列按数值排序，例如预算金额、人天、决算人力、结算人力。
- 日期列按日期排序，例如计划落地日期、预算获取日期、预算承诺日期。
- 文本列按中文本地化文本排序。
- 空值默认排在末尾，升序和降序都不把空值提前。
- 排序入口统一使用表头按钮和 `↕ / ↑ / ↓` 指示器。
- 新增字段必须同步接入排序取值逻辑。

## CSV 导入与映射

- 不假设外部 CSV 字段名、字段顺序、字段数量与页面完全一致。
- 导入 CSV 时先读取表头并推荐自动映射结果。
- 用户必须能手动调整不准确的映射关系，再确认导入。
- 字段映射 UI 使用一个横向滚动表格：第一行是目标字段，第二行是映射下拉框，下面紧跟前 5 条非空数据预览。
- 字段映射和数据预览必须在同一个卡片、同一个横向滚动容器内，不要在映射区和预览区中间再放一个横向拖拉轴。
- 预览数据列宽必须和上方目标字段/映射下拉框保持一致。
- 字段映射控件字体与需求池表格行一致（当前为 12px），下拉框不要过宽。
- 字段名、下拉框内容、预览单元格内容过长时省略展示，鼠标悬停通过 `title` 展示完整内容。
- 字段映射确认前默认展示 CSV 前 5 条非空数据行；如果没有内容则只展示有内容的行/空状态。
- 导入前校验：
  - 必填字段格式
  - 需求单号格式
  - 导入文件内部需求单号重复
  - 与现有需求池需求单号重复
- 导入失败要给出明确错误信息，不允许静默导入 0 条。
- 测试表格应覆盖字段名不同、字段顺序不同、按总量/按分项两类工作量场景。

## 人员配置表格

- 人员配置页面必须复用需求池的基础维护表格风格：居中、12px 表格字体、输入框/下拉框居中、横向滚动、清单不限制高度。
- 人员配置支持和需求池一致的导入/导出体验：读取 CSV 表头、自动推荐字段映射、允许用户修正映射、同表格展示前 5 条非空预览、确认后导入。
- 人员配置中的必填字段也必须带红色 `*`，导入映射和清单表头保持一致。
- 人员类型、角色等枚举字段优先用下拉框，不使用自由输入。

## 表单与详情页

- 详情表单应延续提交需求页的布局、控件和必填标识。
- 新增需求入口不展示仅维护场景使用的字段，例如结算人力；结算人力保留在需求池详情/维护场景。
- 条件字段要按业务状态启用/禁用，例如预算状态为“已获取”时才填写预算获取日期。

## 图表 + 表格页面

- 图表页必须配套明细表，避免只有图形无法追溯。
- 图表空数据时显示明确空状态，不白屏。
- 需求 ID 可点击跳回需求池并展开对应需求。
- 财会/预算类页面优先使用横向条形图、堆叠图、执行率排行等容易对账的形式。
- 预算分析、预算执行、风险分析不得拆成重复页面。
- 预算池、月度占用、需求占用明细、风险规则应集中在统一预算与风险分析页中。
- 图表必须配套可追溯明细表。
- 预算池金额变化和需求池变化必须触发实时重算。
- 需求预算应从匹配预算池中扣减，并展示剩余/超额状态。

## 实施检查清单

修改任何表格页后，至少检查：

1. 表头和单元格是否居中。
2. 小窗口下横向滚动条是否出现在当前表格附近。
3. 展开详情是否仍可横向拖动。
4. 新增字段是否同步到 CSV 模板、导入、导出。
5. 导入映射是否能处理字段名/顺序不一致。
6. 必填项是否有 `*` 标识。
7. 构建是否通过：`npm exec vite -- build`。
