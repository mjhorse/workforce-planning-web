# 人力预算与需求规划平台 Web版操作说明手册

## 1. 系统用途

本系统用于在每月 PO 提交前，帮助需求提交方、预算管理员和开发团队主管完成以下工作：

- 统一收集下月需求清单。
- 配置团队人员数量和汇总成本。
- 配置年度预算、已消耗成本和不同新增预算假设。
- 自动分析不同预算场景下的人力缺口。
- 给出需求保留、延期、削减建议。
- 输出可用于管理沟通的分析报告。

系统不在前台展示单个人成本，只展示人员类型维度的汇总成本。

当前原型版本使用浏览器本地存储保存数据。导入需求、修改预算或修改人员配置后，刷新页面不会丢失；如果更换浏览器或清理浏览器数据，数据会丢失。正式多人版本建议升级为后端数据库。

## 2. 用户角色

### 2.1 需求提交方

主要使用功能：

- 提交需求
- 查看需求池
- 查看自己需求在分析报告中的建议结果

### 2.2 预算管理员

主要使用功能：

- 预算配置
- 预算场景配置
- 查看分析报告中的预算缺口和场景推演

### 2.3 开发团队主管

主要使用功能：

- 人员配置
- 需求池管理
- 分析报告
- 判断哪些需求保留、延期或削减

## 3. 首页仪表盘

### 3.1 功能用途

首页仪表盘用于快速了解当前整体情况，包括：

- 全年预算
- 已消耗成本
- 当前预算结余
- 下月需求总人天
- 未明确预算需求人天
- 不同预算场景下需要减少或延后的人天

### 3.2 操作步骤

1. 打开系统后，默认进入“首页仪表盘”。
2. 查看顶部指标卡：
   - “全年预算”：当前年度已获取的预算总额。
   - “已消耗”：截止当前已经消耗的成本。
   - “预算结余”：预算减去已消耗成本，负数代表已超支。
   - “下月需求”：下月全部需求的总工作量。
   - “未明确预算”：没有明确预算来源的需求工作量。
3. 查看“场景风险概览”：
   - 每个预算场景会显示需要减少或延后的人天。
   - 数字越大，说明该场景下预算压力越大。

## 4. 需求池

### 4.1 功能用途

需求池用于集中管理所有需求提交方提供的下月需求。

可以查看：

- 需求名称
- 需求来源
- 投资来源
- 优先级
- 承接模块
- 版本
- 总工作量
- 是否有预算
- 业务评审结论
- 进展状态

### 4.2 操作步骤

1. 点击左侧菜单“需求池”。
2. 页面展示所有已提交需求。
3. 使用上方筛选框筛选需求：
   - 全部
   - P0 / P1 / P2 / P3 / P4
   - 重要用户
   - 临时紧急需求
   - 无预算
4. 如需批量导入，先点击“下载导入模板”，按模板填写需求数据。
5. 点击“导入CSV”，选择填写好的 CSV 文件，系统会将文件中的需求追加到需求池。
6. 导入完成后，页面会提示成功导入的需求数量。
7. 如需导出，点击“导出CSV”。
8. 如发现错误数据，可点击对应行的“删除”。

### 4.3 注意事项

- 导入 CSV 需要包含表头。建议使用系统下载的导入模板。
- 当前支持中文字段，例如：需求名称、需求来源、投资来源、优先级、版本、计划落地日期、承接模块、总工作量（人天）预估、是否带预算、预算金额w 等。
- 删除操作会从当前页面数据中移除该需求。
- 正式系统中建议增加删除确认和审计日志。

## 5. 提交需求

### 5.1 功能用途

需求提交方通过该页面登记下月需求。

### 5.2 字段说明

| 字段 | 说明 |
|---|---|
| 需求名称 | 简要描述需求内容 |
| 需求来源 | 产品立项规划、重要用户、临时紧急需求 |
| 投资来源 | 财经、行政、HR、制造、ICT、审计、供应、采购、研发、部门自投、无预算等 |
| 优先级 | P0 到 P4，P0 最高 |
| 承接模块 | 按 EAMAP 或团队模块名称填写 |
| 版本 | 期望纳入的版本 |
| 计划落地日期 | 需求计划上线、完成或业务落地的日期 |
| 总工作量 | 总人天预估 |
| 预算金额 | 需求已明确的预算金额，单位 w |
| 当前业务痛点 | 当前业务遇到的问题 |
| 需求目标 | 希望通过需求达成什么目标 |
| 需求价值 | 对业务、用户、合规、效率等方面的价值 |
| 是否带预算 | 是否已有明确预算来源 |
| 是否已承诺 | 是否已经对用户或管理层承诺 |
| 是否可拆分 | 是否可以拆成 MVP 和后续增强 |

### 5.3 操作步骤

1. 点击左侧菜单“提交需求”。
2. 按页面字段填写需求信息。
3. 如果需求已有预算，勾选“是否带预算”，并填写预算金额。
4. 如果需求已经对客户或管理层承诺，勾选“是否已承诺”。
5. 如果需求可以拆分交付，勾选“是否可拆分”。
6. 点击“提交需求”。
7. 系统自动跳转到“需求池”，可以看到刚提交的需求。

## 6. 人员配置

### 6.1 功能用途

人员配置用于维护团队不同人员类型的人数、年度汇总成本和月度可用人天。

系统不展示单个人成本，只展示人员类型汇总成本。

### 6.2 字段说明

| 字段 | 说明 |
|---|---|
| 人员类型 | 自有-作业岗、自有-专业岗、OD、TM |
| 人数 | 当前配置人数 |
| 年度汇总成本w | 该人员类型全年汇总成本，单位 w |
| 月度可用人天 | 该人员类型每月可用于需求承接的人天 |
| 调整口径 | 是否固定基线、是否本次预算调增/削减对象 |

### 6.3 操作步骤

1. 点击左侧菜单“人员配置”。
2. 找到需要调整的人员类型。
3. 修改人数、年度汇总成本或月度可用人天。
4. 修改后系统会自动影响分析报告中的测算结果。

### 6.4 注意事项

- 当前模型默认 TM 是本次预算调增/削减对象。
- 自有和 OD 一般作为基线资源。
- 前台不展示单个人成本，避免敏感信息外显。

## 7. 预算配置

### 7.1 功能用途

预算配置用于预算管理员维护年度预算、已消耗成本和不同新增预算假设。

### 7.2 字段说明

| 字段 | 说明 |
|---|---|
| 全年已获取预算w | 当前年度已批复或已获取预算 |
| 截止当前已消耗成本w | 截止当前团队已经消耗的成本 |
| 月工作日 | 用于人月和人天换算 |
| 年工作日 | 用于后台成本模型换算 |
| 场景名称 | 预算假设名称，例如保守预算、中性预算 |
| 新增预算w | 该假设下预计可新增预算 |
| 说明 | 对该场景的解释 |

### 7.3 操作步骤

1. 点击左侧菜单“预算配置”。
2. 填写或修改全年预算和已消耗成本。
3. 填写月工作日和年工作日。
4. 在“预算场景”表格中配置不同新增预算假设。
5. 修改后进入“分析报告”查看结果。

## 8. 分析报告

### 8.1 功能用途

分析报告用于查看系统基于需求、人力、预算生成的结论。

报告结构包括：

1. 当前预算和需求现状。
2. 后台人员数量和汇总成本配置。
3. 投入分布。
4. 不同预算场景推理。
5. 每个场景下建议保留、延期、削减的需求。

### 8.2 操作步骤

1. 点击左侧菜单“分析报告”。
2. 查看顶部指标，确认当前预算是否超支。
3. 查看“后台配置：人员数量与成本”，确认人员配置是否正确。
4. 查看“投入分布”：
   - 按承接模块查看哪些模块占用人力最多。
   - 按投资来源查看哪些预算来源占用人力最多。
   - 按需求来源查看产品规划、重要用户、临时需求占比。
5. 查看每个预算场景：
   - 新增预算
   - 抵扣后可用预算
   - 可支撑人天
   - 需减少或延后人天
6. 阅读“推理原因”。
7. 查看“建议保留”“建议延期”“建议削减”清单。

### 8.3 如何理解系统建议

#### 建议保留

通常包括：

- P0 / P1 需求
- 重要用户需求
- 合规、审计、监管类需求
- 有预算需求
- 已承诺需求
- 进行中或已排期需求

#### 建议延期

通常包括：

- P2 需求
- 价值明确但预算不足的需求
- 可拆分需求
- 非下月必须交付需求

#### 建议削减

通常包括：

- P3 / P4 需求
- 无预算需求
- 待评审需求
- 价值不清晰需求
- 工作量大但优先级低的需求

## 9. 推荐月度使用流程

### 第一步：需求方提交需求

需求方在每月固定时间前进入“提交需求”，录入下月需求。

### 第二步：开发主管检查需求池

开发主管进入“需求池”，检查需求字段是否完整，必要时要求需求方补充信息。

### 第三步：预算管理员配置预算

预算管理员进入“预算配置”，更新年度预算、已消耗成本和新增预算假设。

### 第四步：开发主管配置人员

开发主管进入“人员配置”，更新不同人员类型的人数、汇总成本和可用人天。

### 第五步：系统生成分析报告

开发主管进入“分析报告”，查看不同预算假设下的需求取舍建议。

### 第六步：管理层决策

基于报告确定：

- 哪些需求必须保留。
- 哪些需求延期。
- 哪些需求削减。
- 是否需要追加预算。
- 下月 PO 保留多少 TM 人力。

## 10. 常见问题

### Q1：为什么系统不展示单个人成本？

为了避免敏感信息外显，系统只展示人员类型维度的年度汇总成本，不展示单个人成本。

### Q2：预算已经超支时，新增预算如何处理？

新增预算会先抵扣历史超支，剩余部分才用于下月 TM PO 测算。

### Q3：系统建议是否等于最终决策？

不是。系统建议用于辅助决策，最终仍需要开发主管和管理层确认。

### Q4：为什么高优先级无预算需求也可能被保留？

P0/P1、重要用户、合规监管或已承诺需求，即使未明确预算，也可能因业务风险较高被建议保留，但需要补充预算来源或管理层豁免。

### Q5：为什么部分有预算需求仍建议延期？

当预算场景下总人天不足时，即使需求有预算，也可能因为优先级较低、非承诺、非下月必须交付而被建议延期。
